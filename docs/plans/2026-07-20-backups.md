# Backups Implementation Plan (Plan 2 of 3)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **Reality check:** most steps run on remote hosts (SSH) or in the Backrest/Komodo web UIs. The human runs those; an agent can write the repo files and verify endpoints.

**Goal:** Every stateful byte on TB and GM lands nightly, restic-encrypted, on Proton Drive (bulk) with a Mega mirror of the critical subset — proven by a restore drill.

**Architecture:** One **Backrest** (restic web UI + scheduler) per Docker node, each deployed as a Komodo stack from this repo, backing up *local* paths and pushing via **rclone** remotes. DB-consistent data comes from nightly **per-service `backup.sh` dumps** run by a host systemd timer. Nothing crosses nodes except the encrypted restic traffic.

**Tech Stack:** Backrest (`garethgeorge/backrest:v1.14.1-alpine` — bundles restic + rclone), rclone `protondrive` + `mega` backends, SOPS-encrypted `rclone.conf`.

## Scope decisions (made in lieu of the deferred brainstorm — veto before executing)

- **One Backrest per node** (TB + GM), not a central orchestrator: restic needs local file access and cross-node mounts (sshfs/NFS) are exactly the fragility we removed. Spec's "Backrest on TB orchestrates" becomes "Backrest on each Docker node, TB holds the pattern".
- **Komodo-scheduled vs backup stack** → backup stack (Backrest's own scheduler). Dumps are per-service `backup.sh` scripts in git, run by a host systemd timer.
- **What is backed up** — TB: Komodo Mongo (dump) + `komodo_keys` volume, Headscale volume (noise key) + sqlite dump, Caddy data volume (certs — avoids ACME rate-limit pain on DR). GM: all of `/home/siffrein/docker-mei/` (cold files today ⇒ file-level backup is consistent). **Not backed up:** the repo + age key (GitHub + password manager), the GM tar.gz archives (they *are* backups, same disk), media (none on these nodes), metrics/logs data.
- **Critical subset (→ Mega mirror)** — TB: DB dumps + `komodo_keys` + headscale data. GM: `actualbudget/`, `n8n/` (finances + workflow exports).
- **Retention:** 7 daily / 4 weekly / 6 monthly, both repos. Prune weekly, `restic check` monthly.
- **Known risk:** rclone's `protondrive` backend is community-maintained and Proton logins can hit captchas/session expiry. T1 verifies it twice before anything depends on it; if it proves flaky in practice, swap the bulk remote for another rclone backend later — repo layout doesn't change.

## Global Constraints

- Never bind `0.0.0.0`: TB services bind `127.0.0.1`, GM services bind `100.64.0.1`.
- Secrets committed sops-encrypted (`secrets.*` / `*.env` per `.sops.yaml`); decrypted files are gitignored; on-node decrypt via the stack's `pre_deploy` (Periphery has `SOPS_AGE_KEY_FILE`).
- New services follow `docs/runbooks/add-a-service.md` (ship dir → stack entry → Caddy handle → sync + deploy). Sync stays **non-prune**; no `[[server]]` blocks.
- Caddyfile changes on TB apply with `docker compose up -d --force-recreate`, never `caddy reload`.
- Container times are UTC — all schedules below are UTC.
- Image tags current at plan time — check for newer patch releases before deploying, don't chase majors mid-plan.

### Task 2: nightly DB dumps on TB (Komodo Mongo + Headscale sqlite)

Dumps run from a **host systemd timer**, not an in-stack sidecar: systemd is the
native scheduler (missed-run catchup, journald logs), it deletes two always-on
busy-sleep containers, and it fits — Periphery already runs as systemd here.
Each service owns its dump (`thriller-bark/<service>/backup.sh`, single
responsibility, colocated); `run.sh` globs `thriller-bark/*/backup.sh` so adding a
stateful service later is just dropping a `backup.sh` in its dir. Everything runs
from the repo checkout, so `git pull` updates the logic with no reinstall. Only
live DBs get a dump; cold volumes (Caddy certs, `komodo_keys`) are backed up raw
by restic in Tasks 3/4.

**Files:**
- Create: `thriller-bark/komodo/backup.sh` (mongodump)
- Create: `thriller-bark/headscale/backup.sh` (sqlite `.dump`)
- Create: `thriller-bark/backups/run.sh` (globs + runs each service's `backup.sh`)
- Create: `thriller-bark/backups/the-sea-dumps.service`
- Create: `thriller-bark/backups/the-sea-dumps.timer`
- Create: `thriller-bark/backups/README.md` (install steps)

**Interfaces:**
- Produces: one dump file per DB on the TB host — `/var/backups/the-sea/dumps/komodo-mongo.archive.gz` and `/var/backups/the-sea/dumps/headscale.sql.gz`, overwritten nightly (restic keeps the history). Task 3 mounts `/var/backups/the-sea` into Backrest; Task 4 backs it up.

Details:
- **Mongo** — `docker exec komodo-mongo` runs `mongodump --archive --gzip` streamed to the host file; the password (`MONGO_INITDB_ROOT_PASSWORD`) stays inside the container, never in the unit or script. mongo:7 ships `mongodump`.
- **Headscale** — host `sqlite3` runs a consistent SQL `.dump` against the named volume (`docker volume inspect -f '{{.Mountpoint}}'`), piped through gzip. The headscale image has no `sqlite3`, so the host provides it; `.dump` is transaction-consistent against the live, container-open db.
- Each `backup.sh` is bash with `set -euo pipefail`, writes `<file>.part` then `mv`, and pipefail aborts before the rename if the dump fails — a bad run keeps the last good file. `run.sh` runs all of them (one failure doesn't stop the rest; overall exit non-zero if any failed). `OnCalendar` is pinned `UTC` (Backrest plans run 04:00 UTC).

Small restores: Backrest UI → snapshot → restore, then copy out of the restore dir.
