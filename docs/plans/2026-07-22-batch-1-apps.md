## Post-benchmark placement revisions

Benchmarks (2026-07-22) overturned the spec's "GM = light node" premise:
- **GM** (OpenVZ, x86 Xeon E5-2670): 8 real cores (**0% steal** under full load),
  32 GB RAM idle, **disk ~95k/41k IOPS**. Weak per-core CPU only.
- **TB** (Oracle ARM Neoverse-N1): ~4├Ś per-core CPU, but disk **throttled to
  ~5.5k/2.4k IOPS** (~17├Ś slower than GM). Mitigated by 14 GB RAM cache.

Rule of thumb going forward: **CPU/edge/sensitive ŌåÆ TB; disk-I/O-heavy / DB / RAM-hungry ŌåÆ GM.**

Two moves from this:
- **Dawarich ŌåÆ GM** (was TB): postgis + a large location-history import is exactly the
  write-heavy workload TB's disk punishes; GM runs it ~17├Ś faster. Location data is in
  Backrest regardless, so GM's lower provider-reliability is an acceptable trade.
- **Observability (Grafana + VictoriaMetrics + Loki) ŌåÆ GM** (was TB, a *live* migration ŌĆö
  sequence deliberately, not part of app batches):
  - Move `thriller-bark/observability/` ŌåÆ `going-merry/observability/`; bind GM's
    `100.64.0.1`, `After=tailscaled`. Datasources stay compose-service-DNS
    (`victoriametrics`/`loki`) since the stack moves together ŌĆö no hairpin-NAT issue.
  - Flip **both** nodes' Alloy push target from `100.64.0.2` ŌåÆ `100.64.0.1`.
  - Caddy `grafana` route ŌåÆ reverse-proxy to `100.64.0.1` instead of TB local.
  - Backups: move `grafana-data` from TB's bulk plan to GM's bulk plan.
  - Reliability caveat: monitoring now dies with GM. Softened by the existing
    Uptime-Kuma on Sunny (external node-liveness). Cap VM/Loki retention regardless.
  - **Follow-up:** update `foundation-design.md` service-distribution table +
    HANDOFF operating rules to match (obs on GM, Dawarich on GM).

## Cross-cutting decisions

### Auth ŌĆö Authelia, not Authentik
- Chosen over Authentik because its config is a **YAML file** ŌåÆ SOPS-encrypt ŌåÆ in
  git, fitting the "everything in git, drop the age key, redeploy" DR model.
  Authentik's config lives in Postgres and can't. Also 1 container vs 4.
- **Single user.** Storage: **SQLite** (local file). Sessions in-memory (no redis).
  `users_database.yml` + config SOPS-encrypted, committed. **Filesystem notifier**
  (no SMTP) ŌĆö reset/2FA-setup links written to a file.
- 2FA: **TOTP** now. **Passkey/WebAuthn deferred** (see future.md) ŌĆö once added at
  Authelia, all OIDC apps inherit it.
**Integration is hybrid** ŌĆö Authelia is *both* an OIDC provider and a forward-auth
gate; method chosen per app by capability. Default policy `two_factor`; SSO via one
shared cookie on `*.siffreinsigy.me` (one login+2FA per session, all apps).

| App | Method |
|-----|--------|
| Dawarich, Open-WebUI, Actual | **OIDC** client (native login button, real identity, no double-gate). Dawarich also bypasses phone-ingestion API paths |
| code-server, Bazarr, Profilarr, Syncthing, Homepage | **forward-auth** (no OIDC support) |
| n8n | forward-auth on editor, **bypass `/webhook` + `/webhook-test`** (OSS n8n has no OIDC) |
| your_spotify | native Spotify login + forward-auth on UI, **bypass OAuth callback + API** |
| Wizarr | native (Plex for users, built-in admin), **bypass invite/join paths** |
| LiteLLM | none ŌĆö internal, API-key, not Caddy-exposed |

Each OIDC app = one client entry (id + hashed secret + redirect URI) in Authelia's
SOPS config; the secret also in that app's SOPS env. All in git.

### Bazarr media access
- Bazarr runs on GM but the media + arr stack live on The Thousand Sunny (can't run
  a 2nd Bazarr there). A prior rclone mount was abandoned only because it went
  stale ŌĆö a maintenance problem, not an incompatibility. FUSE confirmed working in a
  container on GM's OpenVZ kernel (`FUSE_OK` test passed).
- Design: **Komodo-managed `rclone` SFTP mount** ŌåÆ Sunny over the mesh
  (`After=tailscaled` semantics), bind-mounted rw into Bazarr. Komodo's image-update
  polling keeps rclone current ŌĆö fixes the root cause.
- **Audio-sync / Whisper: off** (CPU cost). Provider subtitle downloads only, so
  Bazarr needs the media tree visible to write `.srt`, not to stream whole videos.
- **Task 0 ŌĆö verify GMŌåÆSunny inbound SSH reachability.** The load-bearing, untested
  assumption: Sunny runs **userspace** tailscale and today only makes *outbound*
  mesh connections. SFTP needs GM to connect *into* Sunny's SSH ŌĆö userspace tailscale
  does not bridge inbound tailnetŌåÆhost ports automatically. Test on GM:
  `nc -vz <sunny-mesh-ip> 22`. If it refuses, expose it with `tailscale serve
  --tcp=22 ŌĆ”` on Sunny (or reconsider Bazarr-on-GM entirely). No fallback below
  covers this failure ŌĆö both fallbacks assume the SSH path works.
- **Path mapping (make-or-break).** Sonarr/Radarr report *their* paths; Bazarr sees
  the mount at a different path. Bazarr's path-mapping must translate the two or subs
  land nowhere.
- Secondary risk: mount propagation into the Bazarr container on OpenVZ.
  Fallback: host `systemd` rclone mount (`After=tailscaled`), bind normally.
